Why Your REST API is Lying to You — and How to Fix It
10 June 2026 8 min read 1.2k views Alex Morgan

I've reviewed probably forty APIs built by Nairobi startups over the past two years. Fintech, agritech, healthtech, SaaS. Different industries, different team sizes, different tech stacks. But one mistake appears in nearly all of them, and it's the same mistake: returning HTTP 200 for operations that failed.

Here's what that looks like in practice:

HTTP/1.1 200 OK
Content-Type: application/json

{
  "status": "error",
  "message": "User not found"
}

That response is a lie. The HTTP layer is telling the client everything went fine. The JSON body is quietly admitting it didn't. The client developer now has to check two layers every single time — the HTTP status and a custom status field — and they will inevitably forget one of them at 11pm before a deadline.

Here are the five most common REST API mistakes I see, and exactly how to fix each one.

Mistake 1: Using HTTP 200 for Every Response

HTTP status codes exist for a reason. They are the first signal your client receives, before it even parses the body. Use them correctly and you get error handling for free at the HTTP client level. Ignore them and every consumer of your API has to write custom parsing logic to figure out what actually happened.

The fix is straightforward. Map your outcomes to the right codes:

  • 200 OK — request succeeded, body contains the result
  • 201 Created — resource was created, include a Location header pointing to it
  • 400 Bad Request — client sent invalid data
  • 401 Unauthorized — not authenticated
  • 403 Forbidden — authenticated but not permitted
  • 404 Not Found — resource does not exist
  • 422 Unprocessable Entity — valid syntax, failed business validation
  • 500 Internal Server Error — something broke on your side

Mistake 2: Vague Error Messages

The second most common issue: error messages that tell the client nothing useful. "Something went wrong" is not an error message. "An error occurred" is not an error message. These make debugging a nightmare for anyone trying to integrate with your API.

A useful error response should include a machine-readable error code, a human-readable message, and — for validation errors — a breakdown of exactly which fields failed and why:

{
  "error": {
    "code": "VALIDATION_FAILED",
    "message": "Request validation failed",
    "details": [
      { "field": "email", "issue": "Must be a valid email address" },
      { "field": "phone", "issue": "Must be a Kenyan number starting with +254" }
    ]
  }
}

That structure gives every client developer everything they need to surface a meaningful error to the end user — without calling you on Slack at 2am.

Mistake 3: Inconsistent Response Shapes

I've seen APIs where GET /users returns an array directly, GET /products returns {"data": []}, and GET /orders returns {"orders": [], "total": 0}. Every endpoint is doing something different. Clients have to handle three different shapes for what is essentially the same operation.

Pick one envelope structure and use it everywhere:

{
  "data": { ... },
  "meta": {
    "page": 1,
    "per_page": 20,
    "total": 154
  }
}
Rule of thumb: a client developer should be able to look at one endpoint's response and accurately predict the shape of every other endpoint's response. If they can't, your API is inconsistent.

Mistake 4: No Versioning Strategy

I've seen startups break their mobile app for thousands of users by deploying a breaking API change with no versioning in place. Once a client is in production — especially a mobile app that users haven't updated — you cannot change the contract your API made with it. Not without a versioning strategy.

The simplest approach that actually works in production: version in the URL path from day one. /api/v1/users, /api/v2/users. It's not the most elegant solution, but it's explicit, easy to document, and easy to route at the infrastructure level. Start with v1 even if you think you'll never need v2. You will.

Mistake 5: Missing or Incomplete Documentation

Your API is only as good as its documentation. An API with no docs is not a product — it's a puzzle. At minimum, every endpoint needs: the HTTP method and path, what authentication it requires, what the request body or query parameters look like, what a successful response looks like, and what error codes it can return.

Use OpenAPI (Swagger) and generate the docs from your code. Tools like Postman, Insomnia, and Swagger UI can then render interactive documentation automatically. It takes a few hours to set up and saves weeks of back-and-forth with every developer who integrates with you.

These five fixes won't make your API perfect, but they'll make it honest. And honest APIs are APIs that people actually want to build on. If you're working on an API in Nairobi and want a review, reach out — I do API audits as part of my consulting work.

Building a Dark Mode Toggle That Actually Works
2 June 2026 5 min read Alex Morgan

Every portfolio site tutorial ends with "add a dark mode toggle!" and then shows you how to toggle a class on the <body> element. That works for about five seconds — until the user refreshes the page and the site resets to light mode, or navigates to a new page and sees a blinding white flash before the JavaScript kicks in.

These are fixable problems. Here's the complete approach I use on every site I build.

Step 1: Define your theme with CSS Custom Properties

The foundation is CSS custom properties (variables). Don't hardcode colours anywhere. Every colour value in your stylesheet should reference a variable:

:root {
  --bg-primary: #ffffff;
  --bg-card: #f8f9fa;
  --text-primary: #0a0a0a;
  --text-secondary: #555;
  --accent: #6c63ff;
  --border: #e0e0e0;
}

[data-theme="dark"] {
  --bg-primary: #0a0a0a;
  --bg-card: #12121f;
  --text-primary: #e8e8f0;
  --text-secondary: #9090a8;
  --accent: #6c63ff;
  --border: rgba(255,255,255,0.08);
}

Now switching themes is a single attribute change on the <html> element. No class toggling on individual elements, no JavaScript-driven inline styles. Just document.documentElement.setAttribute('data-theme', 'dark').

Step 2: Persist the preference with localStorage

This is where most tutorials stop short. They show you the toggle but not the persistence. Without this step, every page load resets to the default theme:

const toggle = document.getElementById('themeToggle');
const saved = localStorage.getItem('theme') || 'dark';

// Apply on load
document.documentElement.setAttribute('data-theme', saved);
updateIcon(saved);

toggle.addEventListener('click', () => {
  const current = document.documentElement.getAttribute('data-theme');
  const next = current === 'dark' ? 'light' : 'dark';
  document.documentElement.setAttribute('data-theme', next);
  localStorage.setItem('theme', next);
  updateIcon(next);
});

function updateIcon(theme) {
  const icon = toggle.querySelector('i');
  icon.className = theme === 'dark' ? 'fas fa-sun' : 'fas fa-moon';
}

Step 3: Eliminate the flash of wrong theme (FOUT)

This is the bug that most sites never fix. If your theme JavaScript is in an external file loaded at the bottom of <body>, the page renders in its default theme first, then your script runs and switches it. Users see a flash.

The fix: put a tiny inline <script> in the <head> — before any CSS — that reads localStorage and sets the attribute immediately, before the browser paints anything:

<head>
  <!-- This MUST be before your stylesheet links -->
  <script>
    (function() {
      const t = localStorage.getItem('theme') || 'dark';
      document.documentElement.setAttribute('data-theme', t);
    })();
  </script>
  <link rel="stylesheet" href="css/style.css" />
</head>
Why inline? External scripts are loaded asynchronously or deferred. An inline script in <head> blocks parsing just long enough to set the attribute before the first paint. Keep it small — this is the only time a render-blocking script is the right answer.

Step 4: Respect the OS preference on first visit

If a user hasn't visited your site before, you have no localStorage value. Instead of defaulting to light mode, check what their operating system prefers:

(function() {
  const saved = localStorage.getItem('theme');
  if (saved) {
    document.documentElement.setAttribute('data-theme', saved);
  } else {
    const prefersDark = window.matchMedia('(prefers-color-scheme: dark)').matches;
    document.documentElement.setAttribute('data-theme', prefersDark ? 'dark' : 'light');
  }
})();

New visitors get the theme their OS already chose for them. Returning visitors get their explicit preference. This is how it should work.

The full implementation takes about twenty minutes. The result is a theme toggle that works correctly on first load, persists across pages and sessions, and respects the user's system preference when they visit for the first time. That's what "actually works" looks like.

The State of Web Dev in Kenya: What We're Building in 2026
18 May 2026 6 min read Alex Morgan

Over the past three months I spoke with twenty-three developers working across Nairobi, Mombasa, and Kisumu. I asked them all the same three questions: what are you building, what stack are you using, and where are you stuck? The answers painted a picture of a developer community that is growing fast, solving genuinely hard problems, and hitting a set of very consistent obstacles.

What Kenyan developers are building

The two dominant categories are fintech and agritech, which shouldn't surprise anyone. M-Pesa has been around since 2007 but the ecosystem built on top of it is still expanding. Developers are building payment orchestration layers, merchant dashboards, loan management systems, savings apps, and lending platforms — all deeply integrated with the Safaricom API and, increasingly, with bank APIs via open banking frameworks that are only just arriving.

Agritech is the other major category. The problems being solved are genuinely unique: connecting smallholder farmers with buyers and reducing post-harvest loss, building crop disease detection tools that work on 2G connections, creating SMS-based market price services for farmers who don't own smartphones. These are not problems Silicon Valley has solved. Kenyan developers are solving them from scratch.

What stacks are being chosen

The most popular stack I encountered was React on the frontend, Node.js or Django on the backend, and PostgreSQL for the database. That combination appeared in roughly 60% of the projects I heard about. Java (Spring Boot) is common in larger organisations and financial institutions — mainly because it's what the university curriculum teaches and what the established banks run internally.

Mobile is almost exclusively React Native or Flutter, with a strong preference for Flutter among developers who started building in the last two years. The ability to target both Android and iOS from a single codebase matters enormously when budgets are tight.

Interesting finding: twelve of the twenty-three developers I spoke to are self-hosting on DigitalOcean or Hetzner rather than AWS or GCP. The reason: cost. AWS in the Nairobi region is significantly more expensive than a DigitalOcean droplet in Frankfurt, and the latency is acceptable enough for most use cases.

Where developers are struggling

Three challenges came up in almost every conversation:

  1. Unreliable infrastructure. Kenyan internet is better than it was five years ago, but power and connectivity outages still affect backend reliability. Several developers mentioned building offline-first features not as a product enhancement but as a survival requirement.
  2. Access to senior mentorship. There are more junior developers entering the market than there are senior engineers available to guide them. Multiple developers mentioned spending weeks stuck on problems that a more experienced engineer could have resolved in an hour.
  3. Client education. Several freelancers described spending as much time explaining why good software takes time and costs money as they spend actually building. Clients often compare software quotes to the price of an off-the-shelf template and don't understand the gap.

What's changing fast

AI tooling is arriving quickly. GitHub Copilot and Claude are both widely used among the developers I spoke to — not as a replacement for understanding fundamentals, but as productivity multipliers for boilerplate-heavy tasks. Several developers mentioned that what used to take an afternoon now takes an hour. The developers who are thriving are the ones who understand their domain deeply enough to verify and direct AI output, rather than blindly accepting it.

The Kenyan developer community is also becoming more visible globally. More developers are contributing to open-source projects, speaking at international conferences, and building products that are finding users outside Africa. The talent has always been here. What's changing is the infrastructure — financial, educational, and digital — that allows it to reach its ceiling.

I'll be publishing a deeper breakdown of the fintech stack choices specifically in a follow-up post next month. If you're a Kenyan developer and you'd like to be included in future research, get in touch.

5 PostgreSQL Mistakes I Made as a Junior Developer
3 May 2026 7 min read Alex Morgan

I learned PostgreSQL the hard way: by breaking a live database, once at 6pm on a Friday, and once in front of a client during a demo. Both incidents were entirely my own fault. Here are the five mistakes I made that cost me the most time — and what I do differently now.

Mistake 1: Storing JSON blobs instead of proper relational tables

When I started, JSON columns felt like a superpower. Why design a proper schema when I could just dump a JSON object into a jsonb column and call it done? The answer is: because six months later you need to query inside that JSON, and suddenly every query is slow and the SQL is unreadable.

PostgreSQL's jsonb type is genuinely useful — for truly dynamic, unstructured data. But I was using it as a lazy substitute for proper table design. If you know the structure of your data, model it relationally. Your future self, running a join instead of a jsonb_extract_path, will thank you.

Mistake 2: Ignoring indexes until performance collapsed

For the first few months of a project, queries are fast. The table has a hundred rows. Everything is fine. Then you hit ten thousand rows and suddenly a page that loaded in 80ms is taking four seconds. You look at the query, it looks fine. The problem isn't the query — it's the missing index.

-- This query on a table with 500k rows, no index:
SELECT * FROM transactions WHERE user_id = 1042;
-- Sequential scan: 1.8s

-- After adding the index:
CREATE INDEX idx_transactions_user_id ON transactions(user_id);
-- Index scan: 2ms

The rule I follow now: any column you filter by frequently in a WHERE clause, join on, or order by — index it from the start. Use EXPLAIN ANALYZE regularly to check whether your queries are hitting indexes or doing sequential scans.

Mistake 3: Writing N+1 queries without realising it

This is the one that got me in production. I was fetching a list of orders and then, for each order, running a separate query to get the customer details. Fifty orders meant fifty-one database round trips. At low volume it was invisible. Under real load, it brought the server down.

-- The wrong way (N+1):
SELECT * FROM orders;
-- Then for each order:
SELECT * FROM customers WHERE id = order.customer_id;

-- The right way (single query with JOIN):
SELECT o.*, c.name, c.email
FROM orders o
JOIN customers c ON c.id = o.customer_id;

If you're using an ORM, learn what SQL it's generating. ORMs make N+1 problems very easy to write accidentally. The fix is usually eager loading — fetching related records in the same query rather than lazily loading them one by one.

Mistake 4: Not using transactions for multi-step operations

I once wrote a payment flow that did three things: deduct the buyer's balance, add the seller's balance, and create a transaction record. I wrote them as three separate queries. On a day when the server had a brief connectivity issue, the first query succeeded and the second failed. The buyer's money disappeared into the void.

Anything that involves multiple writes that must all succeed or all fail together belongs in a transaction:

BEGIN;
  UPDATE accounts SET balance = balance - 500 WHERE id = 1;
  UPDATE accounts SET balance = balance + 500 WHERE id = 2;
  INSERT INTO transactions (from_id, to_id, amount) VALUES (1, 2, 500);
COMMIT;
-- If any step fails, ROLLBACK happens automatically

Mistake 5: Skipping database constraints and relying on application logic

My early schema had no foreign keys, no unique constraints, no not-null constraints beyond the obvious. I was enforcing data integrity entirely in my application code. Then a bug created duplicate records. Then a deleted parent record left orphaned children. Then a null slipped into a field that should never be null and caused a crash three weeks later in production.

The database is your last line of defence for data integrity. Application bugs happen. Enforce constraints at the database level:

CREATE TABLE orders (
  id SERIAL PRIMARY KEY,
  user_id INT NOT NULL REFERENCES users(id) ON DELETE RESTRICT,
  status VARCHAR(20) NOT NULL CHECK (status IN ('pending','paid','cancelled')),
  amount NUMERIC(10,2) NOT NULL CHECK (amount > 0),
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

None of these mistakes are unusual for junior developers. What matters is learning them before they cause a production incident — or, if they already have, learning them now so they don't happen again. PostgreSQL is an extraordinarily capable database. Treat it like one from the start.

Was My CS Degree Worth It? An Honest Look Three Years In
14 April 2026 9 min read Alex Morgan

I get asked this question a lot, usually by second or third year students who are staring at a Java assignment and wondering if they could just watch YouTube videos instead. I want to give an honest answer — not a defensive one from someone trying to justify money already spent, and not a cynical one from someone who forgot why fundamentals matter.

Here's what I actually think, three years after graduating from Strathmore University with a BSc in Computer Science.

What the degree taught me that YouTube never could

How to think about problems systematically. The most valuable thing a CS curriculum gives you isn't Java or Python or any specific tool — it's a framework for decomposing complex problems. Algorithms and data structures, discrete mathematics, operating systems theory: these weren't directly relevant to my first job, but they changed how I approached every problem I worked on. I think in time complexity. I notice when a data structure is wrong for a use case. I understand what the computer is actually doing, not just what the framework abstracts away.

How to read technical literature. University forced me to engage with academic papers, technical specifications, and documentation at a depth that I would never have chosen voluntarily. Reading an RFC or a database internals paper without falling asleep is a learnable skill, and I learned it because I had no choice. It's paid off many times over.

How to work under pressure with teams I didn't choose. Group projects at Strathmore were frustrating in exactly the ways that real client projects are frustrating. Miscommunication, unequal effort, last-minute changes, impossible deadlines. That frustration was preparation, even if it didn't feel like it at the time.

What the degree completely failed to cover

The curriculum when I graduated was almost entirely focused on building things, not shipping things. Here's what I had to learn on my own:

  • Version control beyond the basics. I knew what Git was. I did not know how to resolve a merge conflict, rebase a feature branch, write a useful commit message, or review someone else's pull request constructively. These are daily tasks in any professional environment and the degree gave them almost no attention.
  • System design at scale. How do you design a system that handles a million users? How do you think about caching, message queues, database replication, load balancing? The degree covered theory but almost nothing practical.
  • How to estimate, scope, and communicate about technical work. The most important skill for any developer working with clients or non-technical stakeholders is the ability to translate technical decisions into business language. I learned this entirely by making expensive mistakes on real projects.
  • The business side of software. I graduated with no understanding of SaaS pricing, API economics, cloud cost management, or how technical decisions compound into financial outcomes. This matters more than most developers admit.

The honest verdict

The degree was worth it — for me, in the specific way I used it. The theoretical foundation gave me an advantage that I can feel when debugging a complex system problem or architecting something from scratch. The network from Strathmore has sent me real work. The credential opened doors that would have required years of portfolio-building to open otherwise in the Kenyan market.

But the degree alone would not have been enough. The developers I know who are thriving — regardless of whether they have a degree — are the ones who treated formal education as a starting point, not a destination. They built things outside class. They took on freelance work while studying. They read beyond the curriculum. They shipped things and broke things and fixed things.

My actual recommendation for anyone asking me whether to do a CS degree in Kenya in 2026: yes, if you treat it as a foundation and spend the four years also building a portfolio. No, if you think the certificate alone will get you where you want to go. The market is too competitive for that now, and it will only get more so.
